05
Problem Statement

Session Amnesia

Claude Code and all vibe-coding tools have no memory between sessions. Every new session starts cold. The developer spends 20–40 minutes re-establishing context, during which the AI generates plausible-but-wrong suggestions that contradict prior decisions.

  • What architectural decisions were made and why
  • Which patterns are conventions vs. temporary experiments
  • What dependencies were introduced and what problems they solved
  • What was tried and explicitly rejected
  • What open questions remain unresolved and blocking

The Compounding Effect for Solo Founders

A solo founder managing multiple products simultaneously — the exact Keelson target persona — experiences session amnesia multiplied. Working on Lodestar on Monday, Kite on Wednesday, Shuffle on Friday means operating as a team of one with no shared memory between sessions.

Why Existing Solutions Fail

  • CLAUDE.md files — manually maintained, frequently stale, require discipline after every session that doesn't happen
  • Commit messages — capture what changed, not why; not queryable for "what was the decision about auth"
  • README.md — static architecture documentation; doesn't track session-level decisions
  • Claude.ai memory — per-conversation only; not per-project; not version-controlled
06
Solution Definition

Lodestar is an MCP server that, on command, synthesizes the current Claude Code session into a structured .lodestar.md context file. At the next session start, Lodestar loads this file and feeds it as context — giving any AI coding tool a warm start.

The .lodestar.md Schema

interface LodestarContext {
  meta: {
    project: string;        // directory name
    date:    string;        // ISO 8601
    model:   string;        // e.g. claude-sonnet-4-6
  };
  decisions: Array<{
    decision:  string;      // what was decided
    rationale: string;      // why
    files?:    string[];    // affected files
  }>;
  patterns: Array<{
    pattern:  string;       // naming/structural convention
    location: string;       // where it's used
  }>;
  dependencies: Array<{
    package: string;        // npm package name
    purpose: string;        // why added this session
  }>;
  rejected: Array<{
    approach: string;       // what was tried
    reason:   string;       // why rejected
  }>;
  openQuestions: Array<{
    question: string;
    blocking: boolean;
  }>;
  nextSession: string[];    // what to pick up first
}

Product Boundaries

All three products touch memory, context, and knowledge preservation. Without hard boundaries the suite fragments. These are formal boundary commitments — not working hypotheses.

Lodestar
Session state — what was decided this session
Established patterns in this codebase
Dependencies added this session
Rejected approaches and why
Must NOT produce product intent docs → that is Vela's job
Vela
Product intent — what the product is supposed to be
Brief capture after vibe-coding builds
Design intent preservation
Concept drift prevention
Must NOT watch sessions → that is Lodestar's job
Pharos
Historical design decisions across archived projects
Natural language archive retrieval
Long-term institutional memory
500GB+ design file indexing
Must NOT manage live sessions → archive only
Hard Rule Lodestar outputs session state. Vela outputs product intent. These are different ontologies and must never merge. If a Lodestar feature would naturally belong in Vela's domain, stop and route it to Vela. Do not expand Lodestar's scope to absorb it.

Phased Build Plan
1a
Session Synthesizer
MCP tool — on command reads current session file changes and outputs structured .lodestar.md context file. Provider-agnostic (Anthropic, OpenAI, Ollama). lodestar init wizard for first-run setup. lodestar review for progressive disclosure web reader.
Gate: 10 personal sessions · context loads cleanly · no manual editing required
1b
Drift Detector
Upload a brief or prior .lodestar.md; Lodestar compares current session against it and flags decisions that contradict or diverge from original intent.
Gate: Phase 1a stable · 3+ real drift detections confirmed in personal sessions
2
Live Background Agent
Passive watcher that captures context continuously without explicit synthesize commands — fires automatically at session end.
Gate: Phase 1b validated on external users (3+)
3
Cross-Project Diffing
Compare context files across multiple projects; surface recurring patterns, repeated mistakes, and architectural inconsistencies.
Gate: Phase 2 validated · 2+ active projects generating consistent context files
09
lodestar review — Web Reader

A CLI-triggered local HTTP server that opens a browser-based progressive disclosure reader for the current .lodestar.md. It is the primary human-facing surface of Lodestar — and the community candy screenshot.

lodestar review
lodestar review --project /path/to/project
lodestar review --diff   # shows changes vs. previous session

10
Decisions — Locked

These were open questions. They are now decided. Do not reopen them.

✓
.lodestar.md committed or gitignored?
Committed. Version control of decisions is a feature, not a side effect. Add .lodestar.history/ to .gitignore — that directory is local recovery only.
✓
Replace or append on re-synthesis?
Replace with shallow history rotation. .lodestar.md is always overwritten with current session context. Before overwriting, the previous file moves to .lodestar.history/YYYY-MM-DD-HH-MM.md. Maximum 3 history files kept; oldest pruned automatically. Append rejected: appended files degrade into session logs.
✓
Provider-agnostic or Anthropic-only?
Provider-agnostic. Lodestar is community candy for the full vibe-coding audience — Claude Code, Cursor, Windsurf. Locking to Anthropic costs newsletter signups and Keelson audience growth. LLMProvider interface abstracts the synthesis call. Anthropic is the default and recommended provider. OpenAI and Ollama ship as alternatives.
✓
How does the user configure their API key?
Via lodestar init wizard. Key stored in ~/.lodestar.config.json. Never in a project repo, never in mcp.json, never in .env in production. The wizard validates the key before saving and opens the provider's console in the browser if the user doesn't have one.
✓
Does Lodestar cost Ken anything per synthesis?
No. Lodestar runs entirely on the user's machine using their own API key. Ken pays nothing per synthesis. User cost is ~$0.01–0.05 per synthesis — negligible alongside a full AI coding session. Ollama users pay nothing. Zero marginal cost to Titania Labs at any scale.
✓
Token budget for the LLM call?
6,000 tokens max input. If the diff exceeds this, truncate git diff to the most recently modified files first and log a warning. Never silently truncate. For Anthropic/OpenAI, use SDK token counting. For Ollama, estimate at 4 characters per token.
